分块加载大数据

# 前言

当用户网络较差、后端返回大量数据时，如果等待响应体全部接收完毕再响应，可能会导致页面出现卡顿现象、页面内部无法及时更新等。因此在响应体的接收过程中，对响应体已接收到的部分进行呈现，可以极大提高性能，优化性能。

# 普通网络请求

const url = './simulate-data.json'; // 该文件存放大量数据，用于模拟请求大数据（大小7M）
async function loadMusicList_normal(url) {
  const resp = await fetch(url);
  const text = await resp.text();
  console.log(text);
}
loadMusicList_normal(url);

使用这种网络请求方法，在每次发送网络请求时，只有在完全接收完响应体后，才能获得响应体中的数据。

这种方法最大的缺陷就是：如果用户网络环境不好，或者一次性接收大量数据，将会导致页面卡顿（这是使用了 async await），必须等待到数据完全接收完，才会执行后面的代码。

# 网络请求优化：响应体分块加载

const url = './simulate-data.json'; // 该文件存放大量数据，用于模拟请求大数据（大小7M）
async function loadNovel_update(url) {
  const resp = await fetch(url);
  const reader = resp.body.getReader();

  const decoder = new TextDecoder();
  for (;;) {
    const { value, done } = await reader.read();
    if (done) {
      break;
    }
    const text = decoder.decode(value);
    console.log(text);
  }
}
loadNovel_update(url);

# 思路分析

通过 fetch 发起网络请求，在请求发起后的接收数据过程中对响应的流进行暴漏，然后进行加工处理，一直重复循环直到响应体全部接收完成。

# 代码分析（保姆级）

这里我们对上面的代码异步函数 loadNovel_update 进行分析（按照逻辑顺序进行分析，而不是从上到下逐行分析）：

const resp = await fetch(url);

await 操作符用于等待一个 Promise (opens new window) 兑现并获取它兑现之后的值。它只能在异步函数 (opens new window)或者模块 (opens new window)顶层中使用。

阅读上面这段对于 await (opens new window) 的介绍，不难猜出，fetch (opens new window) 的返回值是一个 Promise 。

这里我对 fetch 进行介绍，fetch() 方法用于发起获取资源的请求。它返回一个 promise，这个 promise 会在请求响应后被 resolve，并传回 Response (opens new window) 对象。

所以，这里的 resp 常量为一个 Response 对象。

对于 Response ，我们可以看官方对他的描述：

Fetch API (opens new window) 的 Response 接口呈现了对一次请求的响应数据。

const reader = resp.body.getReader();

由于 resp 为 Response 对象，因此 resp.body 即 Response 对象中的 body 属性。

那么，body 属性是什么呢 ❓ 对此，我们查看官方对此的描述：

一个简单的 getter，用于暴露一个 ReadableStream (opens new window) 类型的 body 内容。

Response (opens new window) 接口的只读属性 body 是暴露响应体内容的 ReadableStream (opens new window)。

值：一个 ReadableStream (opens new window)，或者对于使用空的 body (opens new window) 属性构建的任意的 Response 对象，或没有任何主体 (opens new window)的实际 HTTP 响应 (opens new window)，则为 null (opens new window)。

ReadableStream ：ReadableStream 接口呈现了一个可读取的二进制流操作。

阅读此段话后，可以知道使用 response.body 暴露响应的流，即可以 暴露已接收的响应体的内容 。

这也就达到我们的目的了，在发送网络请求后接收数据时，在响应体数据过大时，对响应体的内容进行分块处理（即在未完全接收全部响应体的数据时，就可以对已接收的数据进行处理）。

response.body 的值是一个 ReadableStream，ReadableStream (opens new window) 接口的 getReader() 方法会创建一个 reader，并将流锁定。只有当前 reader 将流释放后，其他 reader 才能使用。

ReadableStream.getReader()

语法：getReader(mode)

当 mode 参数不指定——缺少值，会返回ReadableStreamDefaultReader (opens new window)，可以从流中返回单个块。

所以 resp.body.getReader() 的返回值为 ReadableStreamDefaultReader，即 reader 的值为 ReadableStreamDefaultReader。

const { value, done } = await reader.read();

在上面，我们分析得到，reader 的值实质上是一个 ReadableStreamDefaultReader。

对于 ReadableStreamDefaultReader ，阅读了官方文档 (opens new window)的描述，对其摘要如下：

Stream API (opens new window) 的 ReadableStreamDefaultReader 接口表示一个用于读取来自网络提供的流数据（例如 fetch 请求）的默认读取器。

ReadableStreamDefaultReader 的其中一个实例方法：ReadableStreamDefaultReader.read() (opens new window)，作用为返回一个 promise，提供对流内部队列中下一个分块的访问权限。

因为 ReadableStreamDefaultReader.read()返回一个 promise，所以我们可以使用 await reader.read() 以获得该 promise 的兑现后的值。

那 ReadableStreamDefaultReader.read()返回的 promise 是什么呢 ❓ 继续阅读文档 (opens new window)，文档对其描述如下：

一个 Promise (opens new window)，其兑现/拒绝的结果取决于流的状态。不同的可能性如下：

  • 如果有分块可用，则 promise 将使用 { value: theChunk, done: false } 形式的对象来兑现。
  • 如果流已经关闭，则 promise 将使用 { value: undefined, done: true } 形式的对象来兑现。
  • 如果流发生错误，promise 将因相关错误被拒绝。

因此，我们可以对返回的该 promise 兑现的值进行解构接收，即 const { value, done } = await reader.read();

这行代码的分析似乎到此就结束了，但 嗯~~~。有细心的人可能发现了上面对 ReadableStreamDefaultReader.read() 作用这里只分析了一半。

这里我们重新看一次这段话：

ReadableStreamDefaultReader.read() (opens new window)，作用为返回一个 promise，提供对流内部队列中下一个分块的访问权限。

对于这后半句话，提供对流内部队列中下一个分块的访问权限，

这时候，我们难免不会疑惑，为什么需要提供对流内部队列中的下一份快的访问权限呢 ❓ 这个访问权限什么时候就被关闭了的呢 ❓

重新回顾上面的代码，可以发现在第二行代码中，我们使用了 resp.body.getReader() 暴漏响应的流，同时也将流进行了锁定。这里对 getReader() 的作用进行回顾：

response.body 的值是一个 ReadableStream，ReadableStream (opens new window) 接口的 getReader() 方法会创建一个 reader，并将流锁定。只有当前 reader 将流释放后，其他 reader 才能使用。

回顾后，兴许你就会豁然开朗，正是 resp.body.getReader() 将流进行锁定了，才需要 ReadableStreamDefaultReader.read() 提供对流内部队列中下一个分块的访问权限。

最后，重新回顾最后获得的值，{ value , done } 。

对于 done ，只有 true 和 false 两个值，但是对于 value ，它的值是什么呢。

那么，这个Uint8Array (opens new window)是什么呢 ❓ 以下为官方文档的描述：

Uint8Array 数组类型表示一个 8 位无符号整型数组，创建时内容被初始化为 0。创建完后，可以以对象的方式或使用数组下标索引的方式引用数组中的元素。

具体展开每个 Uint8Array 中的每次值，发现都是 123、10、32 等的数字，与我们预期的值完全不是一回事。

这里我们使用 TextDecoder 来解析这里的 value，就可以将其转换为我们预期的值，对于 TextDecoder ，官方文档 (opens new window)对其描述如下：

TextDecoder 接口表示一个文本解码器，一个解码器只支持一种特定文本编码，例如 UTF-8、ISO-8859-2、KOI8-R、GBK，等等。解码器将字节流作为输入，并提供码位流作为输出。

const decoder = new TextDecoder() 实例化构造函数，这里我们不指定参数，则默认调用其方法时会将方法接收的参数转换为 utf-8 编码的值。

随后调用 TextDecoder 的方法TextDecoder.prototype.decode() (opens new window)，对于该方法，官方文档 (opens new window)描述如下：

TextDecoder.decode() 方法返回一个字符串，其包含作为参数传递的缓冲区解码后的文本。

解码方法在当前的 TextDecoder (opens new window) 对象中定义。这包含了数据的预期编码，以及如何处理解码时发生的错误。

语法：decode(buffer, options)。buffer、options 可选。buffer 值为一个 ArrayBuffer (opens new window)、TypedArray (opens new window) 或包含要解码的编码文本的 DataView (opens new window) 对象。

这里 buffer 值可以是一个 TypedArray，而上面的 value 打印后的结果Uint8Array (opens new window) 也正是 TypedArray。所以可以将 value 解析成 utf-8 编码的字符串，即预期的格式。

但是，我们还是会很疑惑，为什么使用这个方法，以及为什么这个方法可以将 value 转化为预期值。

在上面，我们直接这样对 Uinit8Array 就直接说是 TypedArray，难免显得很突兀，所以我摘录了一段来自官方文档的话：

# Uint8Array 原型声明 (opens new window)

所有的 Uint8Array 对象继承自 %TypedArray%.prototype (en-US) (opens new window).

而 TextDecoder.prototype.decode() (opens new window) 刚好可以接收一个 TypedArray，因此我们可以将 value 传入该方法作为参数，该方法则发布会将其转化为预期后值。
